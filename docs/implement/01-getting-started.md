# Getting Started

This guide gets you from an empty checkout to a working "hello world" agent run, building **on** the Cove framework rather than re-implementing it. By the end you will have a Convex deployment, the generated bindings, a trivial agent, and a single prompt that runs end-to-end against the reserved `cove-test/mock` model — no provider API key required.

If you want the conceptual map first, read [Overview & Mental Model](00-overview.md). When you're ready to go deeper, see [Defining Agents](02-defining-agents.md) and [Invoking Agents](03-invoking-agents.md).

## Prerequisites

- **Node.js** with `npm` (the project is ESM, `"type": "module"`).
- **A Convex deployment.** Cove is a standalone Convex app — the backend (durable engine, sessions, schema, HTTP) lives under `convex/`. You provision a deployment the first time you run `convex dev` (it provisions a dev deployment and writes `CONVEX_DEPLOYMENT` / `CONVEX_URL` into `.env.local`).
- **(Optional) An AI gateway key** for *live* model calls: `AI_GATEWAY_API_KEY` (a Vercel AI gateway credential, also satisfied by `VERCEL_OIDC_TOKEN`). The gateway key satisfies **any** provider. You do **not** need this for the hello-world below — it uses `cove-test/mock`, which needs no credentials.

> Note: `resolveModel` does **not** gate on credentials. A missing or invalid key only surfaces as an error at request time (the gateway reports it), never at resolve time. To stay credential-free in dev/CI, use `cove-test/mock`.

## Get the code and install deps

The app root is `cove-harness/` (package name `cove`). From the repo root:

```bash
npm install
```

> The bundled `.bin/` shim scripts may lack the `+x` bit (a quirk of macOS-installed `node_modules`). The canonical commands below therefore invoke the bundled binaries **directly with node**. Run everything from the repo root.

## Configure env

Two files hold configuration; both are gitignored, so you supply them yourself after cloning.

`.env.local` carries deployment + client/HTTP wiring:

```bash
# .env.local
CONVEX_DEPLOYMENT=...        # written by `convex dev` on first run
CONVEX_URL=https://<your-deployment>.convex.cloud
CONVEX_SITE_URL=https://<your-deployment>.convex.site   # used by clients / HTTP actions
```

`.env` carries provider + sandbox credentials (only needed for live runs / sandboxed tools):

```bash
# .env
AI_GATEWAY_API_KEY=...       # Vercel AI gateway; satisfies any provider (or set VERCEL_OIDC_TOKEN)
# Optional, situational:
# ANTHROPIC_API_KEY=...
# UPSTASH_BOX_API_KEY=...
# UPSTASH_BOX_NAME=...
```

`CONVEX_URL` is what your SDK client connects to; `CONVEX_SITE_URL` is the base for the HTTP routes (`/agents`, `/runs`, `/channels/slack`). For the SDK-driven hello-world you only need `CONVEX_URL`.

## First deploy (codegen + deploy)

`convex/_generated/` and `.convex/` are gitignored — they are regenerated by Convex. You **must** run one of the commands below after cloning, or `internal.*` / `api.*` references won't resolve and `tsc` will fail.

There are two distinct commands, and the distinction matters:

```bash
# Codegen only — pushes component bindings, regenerates convex/_generated/.
node node_modules/convex/bin/main.js codegen

# Full deploy — also makes your functions callable. Use this.
node node_modules/convex/bin/main.js dev --once
```

`codegen` **only** pushes component bindings; `dev --once` is the **full deploy** that makes functions callable. For getting started, run `dev --once`.

Verify your build typechecks:

```bash
node node_modules/typescript/bin/tsc --noEmit   # must exit 0
```

After `dev --once` succeeds, you can drive the engine directly from the CLI with the dev entry point, using the reserved mock model (free, deterministic, no credentials):

```bash
node node_modules/convex/bin/main.js run dev:startPrompt '{"prompt":"hello"}'
```

`dev:startPrompt` (`convex/dev.ts`) defaults `instanceId` to `"dev"` and `sessionName` to `"default"`, always uses `harnessName: "default"` (it has no `harnessName` argument — the production `submitPrompt` mutation is the one that accepts and defaults `harnessName`), runs with `supersede: false`, and — because no `model` was supplied — defaults to `cove-test/mock`. It returns an `AdmitResult` (`{ sessionId, requestId, submissionId, workflowId }`). Poll the run with `dev:getRequest`:

```bash
node node_modules/convex/bin/main.js run dev:getRequest '{"requestId":"<requestId>"}'
```

When `status` is `"completed"`, `dev:getRequest` reports `finalText` as the byte-stable canned text `"cove mock response"` and `totalTokens` as a flat `9` (the handler exposes `totalTokens` as a number, not a usage object). The full usage object — only returned by `api.requests.get` — is `{ input: 4, output: 5, ..., totalTokens: 9, cost: { ..., total: 0 } }`; note the token total is `totalTokens`, and `total` exists only nested inside `cost`.

## Hello world: define an agent and run one prompt via the SDK

This is the real entry point for building on Cove. We'll define a trivial agent, then drive it through the Convex-native SDK (`createCoveClient`) end-to-end.

### 1. Define and register the agent

`createAgent` lives in the pure runtime barrel. `defineAgentRegistry` / `registerAgentRegistry` are **Convex-app-bound** and import from `convex/`, not the `@cove/runtime` barrel.

The `createAgent` initializer runs **every time** the runtime initializes a harness from the agent (it is not a one-time constructor). It returns an `AgentRuntimeConfig` — note there is **no** top-level `name` field on the config (an agent is addressed by its registry key), and **no** `systemPrompt` field (authors set `instructions`; the system prompt is composed internally).

```ts
// convex/agents.ts
import { createAgent } from "../src/runtime/agent-definition.ts";
import { defineAgentRegistry, registerAgentRegistry } from "./agentRegistry.ts";

// A trivial agent. `model: "cove-test/mock"` makes runs free, deterministic,
// and credential-free. instructions becomes part of the composed system prompt.
export const helloAgent = createAgent(() => ({
  model: "cove-test/mock",
  description: "A minimal hello-world agent.",
  instructions: "You are a friendly assistant. Greet the user.",
}));

// Agents have no filesystem addressing on Convex — register an explicit
// name -> createAgent() map. Names must match /^[A-Za-z][A-Za-z0-9_-]*$/.
export const registry = defineAgentRegistry({
  hello: helloAgent,
});

// Install the registry so setup can resolve agents by name.
// (The generated app entry re-installs this on each cold boot.)
registerAgentRegistry(registry);
```

> The `model` field is a `'provider-id/model-id'` string, or `false` to force every model-using call to pass a call-level `options.model`. Here we hardcode `cove-test/mock` so the agent's default model is the mock.

> **Registries and codegen.** `cove init` scaffolds `convex/agentRegistry.ts` alongside `convex/toolRegistry.ts` and `convex/extensionRegistry.ts` (for declaring custom tools and extensions by name). `cove build` then codegens the matching resolvers under `convex/_cove/` — `agentResolver.ts`, `toolResolver.ts`, and `extensionResolver.ts` — which install each registry so `setup` can resolve agents, tools, and extensions by name. (`cove dev` runs the same codegen.)

Redeploy so the new functions are bundled and callable:

```bash
node node_modules/convex/bin/main.js dev --once
```

### 2. Drive it with the SDK

`createCoveClient(client, refs, opts?)` wraps any object satisfying `ConvexLike` (both `ConvexHttpClient` and `ConvexReactClient` qualify — it only needs `mutation(ref, args)` and `query(ref, args)`). You supply the **5 deployed function references** yourself via `CoveApiRefs`, because the SDK is intentionally decoupled from `convex/_generated`.

```ts
// scripts/hello.ts
import { ConvexHttpClient } from "convex/browser";
import { api } from "../convex/_generated/api";
import { createCoveClient } from "../src/sdk/index.ts";
import { helloAgent } from "../convex/agents.ts";

const client = new ConvexHttpClient(process.env.CONVEX_URL!);

// All 5 refs are required. Pass them from the app's generated `api`.
const cove = createCoveClient(client, {
  submitPrompt: api.invoke.submit.submitPrompt,
  stopActive: api.invoke.submit.stopActive,
  getRequest: api.requests.get,
  sessionExists: api.sessions.store.exists,
  deleteSession: api.sessions.store.remove,
});

// Authoring chain: context -> init(agent) -> session -> prompt.
const ctx = cove.context({ id: "hello-instance" });   // only `id` is required
const harness = await ctx.init(helloAgent);            // resolves defaultModel from the agent profile
const session = await harness.session();               // sessionName defaults to "default"

const { text, usage, model } = await session.prompt("Say hello.");

console.log(text);   // "cove mock response"
console.log(model);  // { provider: "cove-test", id: "mock" }
console.log(usage);  // token + cost usage for the run
```

Run it (with `CONVEX_URL` exported from `.env.local`):

```bash
node --experimental-strip-types --env-file=.env.local scripts/hello.ts
```

What just happened, mapped to the API:

- **`cove.context({ id })`** builds a `CoveContext` for one invocation. Only `id` is required; `payload` defaults to `undefined`, `env` to `{}`, and the logger to a no-op.
- **`ctx.init(helloAgent)`** calls the agent's `initialize`, resolves its profile, and derives the harness `defaultModel` from `profile.model` (`"cove-test/mock"` here). Pass `{ name }` to set the harness name (defaults to `"default"`).
- **`harness.session()`** returns a session named `"default"`. (It validates the name but does **not** check existence — use `harness.sessions.get()` / `.create()` if you need existence semantics.)
- **`session.prompt(text)`** is the core call. Model resolution is `options.model ?? harness defaultModel`; with neither set it throws `ModelNotConfiguredError`. It returns a `CallHandle` — a thenable you `await` (it also exposes `.signal` and `.abort(reason?)` for cancellation).
- Awaiting a plain prompt resolves to **`{ text, usage, model }`**. `text` is the run's `finalText`; `model` is parsed from the model string by splitting on the first `/` (so `"cove-test/mock"` becomes `{ provider: "cove-test", id: "mock" }`).

Under the hood the SDK submits via the `submitPrompt` mutation and then **polls** `api.requests.get` until the request reaches a terminal status (`completed` / `failed` / `cancelled`). There is no streaming/SSE — polling defaults to a 400ms interval with a 600,000ms (10 min) deadline, both tunable via the third `CreateCoveTransportOptions` argument to `createCoveClient`.

### Typed results (optional)

To get a re-validated typed value back instead of raw text, pass a valibot schema as `options.result`:

```ts
import * as v from "valibot";

const { data } = await session.prompt("Return a greeting.", {
  result: v.object({ greeting: v.string() }),
});
// `data` is the typed, re-validated output.
```

The schema is converted to JSON Schema for the server, then the captured value is **re-validated locally** with valibot before resolving. You either get validated `data` or a thrown `ResultUnavailableError` — never unvalidated data. (Note: the mock model returns fixed text, so a strict result schema will fail re-validation against it; use a real model when exercising result schemas.)

## What's wired vs. deferred

Be aware that this is an early scaffold. In the current cut, **only `session.prompt()` is implemented** through the SDK facade. The following reject with a `CoveError` whose code is `'not_implemented'`:

- `session.skill(...)`, `session.task(...)`, `session.shell(...)`, `session.fs`, `session.compact()`
- `harness.shell(...)`, `harness.fs`

Channel **inbound** (the Slack webhook) is implemented, but outbound reply posting is not. `POST /workflows/:name` is a live route that currently always returns `404 workflow_not_found`. These gaps are called out in the relevant guides rather than implied to exist.
