# Deployment & Operations

This page is the operations runbook for a developer building **on** Cove — wiring your agents and workflows into the `cove-harness` Convex app, deploying it, configuring credentials, and keeping a clean build. It does not re-derive the engine internals; it covers the commands you run, the env vars you set, and the project rules you must respect.

If you're brand new, start with [Getting Started](01-getting-started.md). For the authoring APIs referenced here, see [Defining Agents](02-defining-agents.md), [Invoking Agents](03-invoking-agents.md), and [Subagents & Workflows](06-subagents-and-workflows.md).

> **One thing to internalize first:** the canonical CLI commands invoke the bundled binaries through `node` directly (e.g. `node node_modules/convex/bin/main.js dev --once`). The `.bin/` shim scripts in `node_modules` lack the executable (`+x`) bit on macOS-installed dependencies, so the bare `convex` / `tsc` / `vitest` commands may fail. The `node node_modules/...` form always works. The `package.json` `scripts` (`convex`, `deploy`, `typecheck`, `test`) exist, but `.rules` mandates the direct invocations.

## 1. The deploy flow: `codegen` vs `dev --once`

There are two Convex commands you'll run constantly, and the difference between them is load-bearing.

```bash
# Codegen ONLY — pushes component bindings and regenerates convex/_generated/.
# Does NOT make your functions callable.
node node_modules/convex/bin/main.js codegen
# (also reachable as: npx convex codegen)

# Full deploy — pushes everything and makes api.*/internal.* functions callable.
node node_modules/convex/bin/main.js dev --once
```

From `.rules` (lines 18-19), the rule is exact:

> `codegen` only pushes component bindings; `dev --once` is the full deploy that makes functions callable.

What this means in practice:

- **After cloning, or after deleting `convex/_generated/`**, you MUST run `codegen` or `dev --once` before anything else. `convex/_generated/` and `.convex/` are **gitignored** (regenerated by `convex dev` / `convex codegen`), so on a fresh checkout the generated `api` and `internal` modules don't exist yet. Until you regenerate them, every `import { api } from "./_generated/api"` is unresolved, your agent/workflow registrations won't link, and `tsc` will fail.
- **`codegen` is enough to get types passing** (it regenerates the bindings TypeScript needs), but it does **not** deploy your functions — calling `api.invoke.submit.submitPrompt` against the deployment will still 404.
- **`dev --once` is the real deploy.** It pushes the schema (`convex/schema.ts`), registers the `@convex-dev/workflow` component (`convex/convex.config.ts`), and makes every query/mutation/action/HTTP route callable.

A typical first-run sequence after `git clone` + `npm install`:

```bash
node node_modules/convex/bin/main.js dev --once      # full deploy (also regenerates _generated/)
node node_modules/typescript/bin/tsc --noEmit        # confirm types resolve (exit 0)
node node_modules/vitest/vitest.mjs run              # run the test suite
```

For production releases, Convex's deploy command is wired as an npm script (`"deploy": "convex deploy"`), and `CONVEX_DEPLOY_KEY` is the credential it uses (see [§2](#2-environment-variables)). The day-to-day iteration loop, though, is `dev --once`.

### Driving a function after deploy

Once `dev --once` has run, you can exercise functions from the CLI with `convex run`:

```bash
# Drive the durable engine end-to-end with the FREE, deterministic mock model.
node node_modules/convex/bin/main.js run dev:startPrompt '{"prompt":"hello"}'
```

`dev:startPrompt` (`convex/dev.ts`) is the minimal dev entry. It defaults `instanceId` to `"dev"`, hardcodes `harnessName` to `"default"`, defaults `sessionName` to `"default"`, and runs with `supersede: false` (it does **not** cancel in-flight work — unlike the production `api.invoke.submit.submitPrompt`, which always supersedes). It returns an `AdmitResult` (`{ sessionId, requestId, submissionId, workflowId }`). Because `dev:startPrompt` omits `model`, the run uses the default `cove-test/mock` model — a free, no-credential, byte-stable run. Poll it with:

```bash
node node_modules/convex/bin/main.js run dev:getRequest '{"requestId":"<requestId>"}'
```

`dev:getRequest` is richer than the public `api.requests.get` — it returns per-step detail (`steps: { stepNumber, isFinalized, finishReason, text, toolCallCount }[]`). Convex functions **cannot be unit-tested** in this repo (no `convex-test` installed), so `dev --once` + `convex run` is how you verify the Convex glue end-to-end. See [§5](#5-the-cove-testmock-seam-deterministic-free-runs) for more on the mock model.

## 2. Environment variables

Cove reads no env vars in the pure runtime (`src/runtime/*`) or the SDK facade (`src/sdk/index.ts`) — configuration there is entirely via the values you pass in code. Env vars matter in two places: **local tooling** (the `.env*` files the CLI reads) and the **deployed Convex backend** (where the durable engine actions and HTTP routes read `process.env`).

### Where each var lives

Both `.env` and `.env.local` are **gitignored** — and so is every `.env.*.local` variant. **They are never committed.** You supply them yourself after cloning.

| File | Variables | Purpose |
| --- | --- | --- |
| `.env.local` | `CONVEX_DEPLOYMENT`, `CONVEX_URL`, `CONVEX_SITE_URL` | Which deployment the CLI targets; `CONVEX_URL`/`CONVEX_SITE_URL` are for clients and HTTP actions. |
| `.env` | `AI_GATEWAY_API_KEY` (or `VERCEL_OIDC_TOKEN`), `UPSTASH_BOX_API_KEY`, `UPSTASH_BOX_NAME`, optional `ANTHROPIC_API_KEY`, `CONVEX_DEPLOYMENT`, `CONVEX_DEPLOY_KEY` | LLM gateway + sandbox credentials and deploy keys. |

### Credentials for live LLM calls

For real model calls you need real provider credentials. The simplest path:

- **`AI_GATEWAY_API_KEY`** (or **`VERCEL_OIDC_TOKEN`**) — the Vercel AI Gateway credential. It satisfies **any** provider through the gateway, so a single key unlocks Anthropic, OpenAI, Google, etc.
- Provider-specific keys also work: `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` for those providers; Google accepts `GOOGLE_API_KEY` / `GEMINI_API_KEY` / `GOOGLE_GENERATIVE_AI_API_KEY` / `GOOGLE_APPLICATION_CREDENTIALS`; Bedrock accepts the AWS key pair.

Important nuance: **`resolveModel` does not gate on credentials.** A missing or invalid key does not fail at model-resolve time — the gateway surfaces a credential error only at request time. `hasCredentialsFor(providerId)` (`convex/providers/gateway.ts`) reads creds via plain `process.env` and is exported for setup diagnostics, but the engine never blocks on it. The diagnostic mapping: gateway creds satisfy any provider; `anthropic`/`openai` also accept their `*_API_KEY`; `google` accepts the four Google vars; `bedrock` accepts the AWS key pair or chain.

To avoid needing any credentials at all in dev/CI, use the `cove-test/mock` model ([§5](#5-the-cove-testmock-seam-deterministic-free-runs)).

### Channel secrets

If you wire up the Slack inbound webhook (see [Channels](07-channels.md)), the route `POST /channels/slack` reads **`SLACK_SIGNING_SECRET`** from `process.env` inside the handler. If unset, every non-challenge request returns `401 UnauthorizedError` (`[cove] Slack signing secret not configured.`).

### Setting env vars on the deployed backend

Local `.env`/`.env.local` cover the CLI. Values your deployed functions read at runtime (`AI_GATEWAY_API_KEY`, `SLACK_SIGNING_SECRET`, provider keys, `UPSTASH_BOX_*`) must be set on the **Convex deployment**. Two ways:

```bash
# Set an env var on the deployment from the CLI:
node node_modules/convex/bin/main.js env set AI_GATEWAY_API_KEY sk-...
node node_modules/convex/bin/main.js env set SLACK_SIGNING_SECRET <slack-secret>

# List / inspect:
node node_modules/convex/bin/main.js env list
```

Or set them in the **Convex dashboard** under your deployment's *Settings → Environment Variables*. Dashboard and `convex env set` write to the same place — use whichever fits your workflow. (`node node_modules/convex/bin/main.js env ...` is the general form for env management.)

## 3. Project layout & the one-way dependency rule

`cove-harness/` is a single standalone Convex app (`package.json` name `"cove"`, `"type": "module"`, ESM, private, `v0.0.0`). It has a strict **three-layer** layout with a **one-way** dependency rule.

```
cove-harness/
├── src/
│   ├── runtime/        # portable, pure, V8-safe logic — the @cove/runtime surface
│   │                   #   barrel: src/runtime/index.ts
│   │                   #   imports NONE of: ai/@ai-sdk/*, @upstash/box, node builtins, Convex
│   └── sdk/            # Convex-native consumer client (createCoveClient) — src/sdk/index.ts
├── convex/             # the Convex backend: schema, durable engine, sessions,
│                       #   invoke, providers, channels, HTTP. MAY import src/runtime/*.
├── docs/
├── package.json
├── tsconfig.json
└── .env / .env.local   # gitignored
```

### The three layers

- **`src/runtime/`** — Portable, pure, V8-safe logic. This is the public `@cove/runtime` surface; the barrel is `src/runtime/index.ts`. The authoring API lives here: `createAgent`, `defineAgentProfile`, `defineTool`, `createCoveContext`, and the full type contract. It imports **none** of `ai`/`@ai-sdk/*`, `@upstash/box`, node builtins, or Convex.
- **`convex/`** — The Convex backend (system-of-record schema, durable engine, sessions, invoke, sandbox, providers, events, HTTP). `convex/*.ts` **MAY** import `src/runtime/*`, but `src/runtime/*` **MUST NEVER** import from `convex/`.
- **`src/sdk/`** — The Convex-native consumer client (`createCoveClient` over a `ConvexLike` client). See [Invoking Agents](03-invoking-agents.md).

### The rule, stated plainly

> `convex/*` may import `src/runtime/*` (verbatim, with `.ts` extensions). `src/runtime/*` must **never** import from `convex/`.

A direct consequence you'll hit while authoring: the **Convex-app-bound** constructs are exported from `convex/`, **not** from the `@cove/runtime` barrel:

- `defineAgentRegistry`, `registerAgentRegistry` → `convex/agentRegistry.ts`
- `defineWorkflow`, `defineWorkflowRegistry`, `registerWorkflowRegistry` → `convex/workflowRegistry.ts`

`createAgent` / `defineAgentProfile` / `defineTool` / `createCoveContext` come from `@cove/runtime` (re-exported via `src/runtime/index.ts`). So a registry file imports from both sides:

```ts
import { createAgent } from "../src/runtime/index.ts";          // pure runtime
import { defineAgentRegistry } from "./agentRegistry.ts";       // Convex-app-bound

export default defineAgentRegistry({
  assistant: createAgent((ctx) => ({
    model: "anthropic/claude-sonnet-4-5",
    instructions: "You are a helpful assistant.",
  })),
});
```

(Agents have no filesystem addressing — you register an explicit `name -> createAgent()` map. Names must match `/^[A-Za-z][A-Za-z0-9_-]*$/`. See [Defining Agents](02-defining-agents.md).)

### The `"use node"` sub-rule

The one-way rule has a runtime companion: only files whose **first statement** is `"use node"` may import `ai`/`@ai-sdk/*`, `@upstash/box`, or node builtins — and those are reached **only** from `"use node"` engine actions, never from queries/mutations or the V8-safe core.

```ts
"use node";                       // MUST be the first statement
import { resolveModel } from "./gateway.ts";   // pulls @ai-sdk/* — only legal under "use node"
```

Queries and mutations stay AI-SDK-free and box-free; only `"use node"` actions touch the sandbox or AI SDK. (`convex/channels.ts` and `convex/http.ts` deliberately avoid `"use node"` — their HTTP actions only call mutations/queries.)

## 4. Typecheck (`tsc`) and tests (`vitest`)

### Typecheck

```bash
node node_modules/typescript/bin/tsc --noEmit
```

This **must exit 0 on every change** (`.rules` line 16). Notable `tsconfig.json` settings that shape how you write imports:

- `moduleResolution: "Bundler"` with `allowImportingTsExtensions: true` — intra-repo imports carry the literal `.ts` extension (e.g. `import { admitPrompt } from "./invoke/admit.ts"`).
- `verbatimModuleSyntax: true` and `isolatedModules: true` — be explicit with `import type` for type-only imports.
- `strict: true`, `types: ["node"]`.
- `include` is `["src/**/*.ts", "convex/**/*.ts"]`; `exclude` is `["node_modules"]`. Crucially, that means **test files under `__tests__/` ARE typechecked** (they live under `src/**` / `convex/**`).

Reminder from [§1](#1-the-deploy-flow-codegen-vs-dev---once): if `tsc` fails with unresolved `./_generated/api` or `./_generated/server` imports, you haven't run `codegen`/`dev --once` since the last clean checkout. Regenerate first.

### Tests

```bash
node node_modules/vitest/vitest.mjs run            # whole suite
node node_modules/vitest/vitest.mjs run convex/engine/__tests__/foo.test.ts   # one file
```

Test conventions (`.rules` lines 3-13):

- Tests live in a **`__tests__/` folder beside the code they test** — never co-located/blended into a source module. Example: `convex/engine/foo.ts` → `convex/engine/__tests__/foo.test.ts`.
- The `__tests__` name starts with `_`, so the **Convex bundler auto-ignores it** (just like `_generated`) — test files never ship as deployed functions. Vitest still discovers `**/__tests__/**/*.test.ts`, and `tsc` still typechecks them.
- **Convex functions cannot be unit-tested** here (no `convex-test` installed). Test the **pure injectable cores** (the logic in `src/runtime/*` and the pure helpers under `convex/`) with vitest, and verify the Convex glue via `dev --once` + `convex run`.
- Drive engine tests deterministically through the in-process mock seam (`cove-test/mock`), never by importing `ai/test` (see [§5](#5-the-cove-testmock-seam-deterministic-free-runs)).

## 5. The `cove-test/mock` seam: deterministic, free runs

The model specifier **`cove-test/mock`** (`RESERVED_TEST_MODEL_ID`, in `convex/providers/testModel.ts`) is the key operational tool for dev and CI. It resolves — via the mock seam, which `resolveModel` checks **first**, bypassing the provider/model slash split — to an in-process `MockLanguageModelV2` `ModelHandle` that needs **no live provider and no credentials**.

What it gives you:

- **Byte-stable output.** Every default mock turn returns the canned text `"cove mock response"` (`RESERVED_TEST_MODEL_TEXT`) with fixed usage `{ inputTokens: 4, outputTokens: 5, totalTokens: 9 }`. Two resolves produce byte-identical output, so replay-equality is exact.
- **The default model at admission.** When no model is supplied, several admission points default to `cove-test/mock` (`convex/invoke/admit.ts` uses `args.model ?? "cove-test/mock"`; `convex/engine/setup.ts` uses `request.model ?? session.model ?? "cove-test/mock"`; `dev:startPrompt` omits `model` and so inherits this default). Pass `model` explicitly for production runs.

Helpers exported from `convex/providers/testModel.ts` for stubbing in tests:

- `isTestModelId(modelString)` — true iff the string is `"cove-test/mock"`.
- `makeMockLanguageModel(options?)` — build a `LanguageModelV2` mock; override `doGenerate`/`doStream` to stub behavior.
- `makeDefaultMockModel()` — the byte-stable default (equals `makeMockLanguageModel()`).
- `makeTestModelHandle(mock?, options?)` — a full `ModelHandle` (`contextWindow 200_000`, `maxOutputTokens 64_000`, zero cost).

> **Never import `ai/test` to get a mock.** That subpath statically pulls in `msw`, an unshipped devDependency — it would require a phantom dependency the consumer install does not carry. Cove hand-rolls the `LanguageModelV2` directly in `testModel.ts` instead.

End-to-end smoke with zero credentials:

```bash
node node_modules/convex/bin/main.js dev --once
node node_modules/convex/bin/main.js run dev:startPrompt '{"prompt":"ping"}'   # uses cove-test/mock
```

## 6. Operational notes

### Regenerating `convex/_generated/`

`convex/_generated/` and `.convex/` are gitignored — they are **build artifacts**, regenerated by `convex dev` / `convex codegen`. Operational implications:

- **Treat them as derived, not source.** Don't edit them; don't commit them; don't expect them in a fresh clone.
- **Regenerate after a clone, a dependency bump, or a schema/function change** to keep `internal.*` / `api.*` references resolving and `tsc` green. `codegen` is the minimum (bindings only); `dev --once` does it as part of a full deploy.
- The generated `api` is also what your SDK consumers depend on. The SDK package is intentionally decoupled from `convex/_generated` — `createCoveClient(client, refs, opts)` requires you to **pass the function references yourself** via `CoveApiRefs` (the five deployed refs: `submitPrompt`, `stopActive`, `getRequest`, `sessionExists`, `deleteSession`, sourced from your app's generated `api`, e.g. `api.invoke.submit.submitPrompt`). See [Invoking Agents](03-invoking-agents.md).

### Regenerating the `convex/_cove/` resolvers

Separate from Convex's own `convex/_generated/`, the `cove` CLI maintains a second set of sidecar files under `convex/_cove/`. `cove build` (and `cove dev`) read your registry exports and **codegen one resolver per registry** — `agentResolver.ts`, `workflowResolver.ts`, `toolResolver.ts`, and `extensionResolver.ts` — each of which installs its registry on import (the cold-boot re-install described in [§6](#module-scoped-state-must-be-re-applied-per-cold-boot)). `cove init` scaffolds the source registries themselves (`agentRegistry.ts`, `workflowRegistry.ts`, `toolRegistry.ts`, `extensionRegistry.ts`) so a fresh project has all four. Treat `convex/_cove/` like `convex/_generated/`: derived, not source — regenerate it after adding or renaming an agent, workflow, tool, or extension.

### `.env` is never committed

`.env`, `.env.local`, and all `.env.*.local` variants are in `.gitignore`. Anyone cloning the repo must supply their own. The split is by audience:

- `.env.local` → deployment targeting for the CLI (`CONVEX_DEPLOYMENT`, `CONVEX_URL`, `CONVEX_SITE_URL`).
- `.env` → secrets and provider/sandbox credentials (`AI_GATEWAY_API_KEY` / `VERCEL_OIDC_TOKEN`, `UPSTASH_BOX_API_KEY`, `UPSTASH_BOX_NAME`, optional `ANTHROPIC_API_KEY`, `CONVEX_DEPLOY_KEY`).

Remember that runtime secrets your **deployed** functions read (the gateway key, `SLACK_SIGNING_SECRET`, provider keys, `UPSTASH_BOX_*`) must also be set on the deployment via the dashboard or `convex env set` — local `.env` does not propagate to the running backend ([§2](#2-environment-variables)).

### The durable-workflow component

`convex/convex.config.ts` registers the `@convex-dev/workflow` component that backs the agent loop. Each per-turn unit of work (LLM streaming + tool dispatch) runs as durable workflow steps, so a crash or redeploy resumes from the last committed step rather than restarting the turn. You don't operate this directly, but it's why `dev --once` (which pushes the component) is the deploy boundary, and why a redeploy mid-run is safe.

### Module-scoped state must be re-applied per cold boot

Some installation hooks are module-scoped and **last-write-wins**, and are **not persisted across cold boots** — the generated app entry must re-apply them on every boot:

- `registerAgentRegistry(registry)` / `registerWorkflowRegistry(registry)` / `registerToolRegistry(registry)` / `registerExtensionRegistry(registry)` — install the active registries that `setup` (and the `POST /workflows/:name` route) resolve agents, workflows, tools, and extensions against. The codegened `convex/_cove/*Resolver.ts` files (below) do these re-installs for you.
- `configureAuthorize(hook)` — installs the HTTP authorize hook for `/agents` and `/runs`. **With no hook installed, those routes are open (unauthenticated).** See [Invoking Agents](03-invoking-agents.md) for gating HTTP access.
- `configureErrorRendering({ devMode })` — toggles whether `500` error details leak to clients (default redacted).

## 7. Documented gaps (don't rely on these yet)

Be honest with yourself and your users about what isn't wired in this cut:

- **SDK facade — only `prompt()` is implemented.** `session.skill` (P10), `session.task` (P6), `session.shell` (P6), `session.fs` (P6), `session.compact` (P12), `harness.shell` (P6) and `harness.fs` (P6) all reject with a `CoveError` whose code is `'not_implemented'`. See [Invoking Agents](03-invoking-agents.md).
- **`POST /workflows/:name` is a stub.** The route and its `CoveApiError` envelope are live, but it always returns `404 workflow_not_found` until the codegen for the workflow registry lands (P8.5). Drive workflows via the native path for now. See [Subagents & Workflows](06-subagents-and-workflows.md).
- **Slack is inbound-only.** The `POST /channels/slack` route maps an event to `api.invoke.submit.submitPrompt` but does **not** post the run result back to Slack — outbound reply is the documented P11 remainder and is not implemented. See [Channels](07-channels.md).
- **No MCP wiring (P10).** `connectMcpServer` appears only in planning/design docs (Skills + MCP, P10) and is **not** present in `convex/` or `src/` in this cut. Do not import or rely on it yet.
- **No streaming / SSE anywhere.** `GET /runs/:runId` and `api.requests.get` are point-in-time snapshots. For live updates use the reactive Convex query `useQuery(api.requests.get, { requestId })`, or HTTP long-poll via `POST /agents/...?wait=result` (60s deadline, 400ms interval).
- **Early scaffold.** `package.json` is `v0.0.0`, private; `src/sdk` is a later phase (P9). Expect surfaces to fill in over time.
