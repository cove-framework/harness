# G2.4 — CLI + codegen (`cove` binary)

> *Completes [P8.5](../group-1/phase-08.5-cli-codegen.md). Group-1 landed the registry **constructs** and their seams but left them **orphaned**: [`convex/agentRegistry.ts`](../../../convex/agentRegistry.ts) ships `defineAgentRegistry`/`registerAgentRegistry`/`getRegisteredAgent` and [`convex/workflowRegistry.ts`](../../../convex/workflowRegistry.ts) ships `defineWorkflow`/`defineWorkflowRegistry`/`registerWorkflowRegistry`/`getRegisteredWorkflow`, yet `grep -rn registerAgentRegistry convex/ src/` outside the two registry files returns **only** [`convex/__tests__/registries.test.ts`](../../../convex/__tests__/registries.test.ts) — nothing installs a registry into a running app. [`convex/engine/setup.ts`](../../../convex/engine/setup.ts) still resolves a model from `request.model ?? session.model ?? "cove-test/mock"` (line 31), **never** `getRegisteredAgent(name)`. [`convex/http.ts`](../../../convex/http.ts) `POST /workflows/:name` (lines 131–139) is a hard-404 stub returning `WorkflowNotFoundError`. There is no `bin/`, no `"bin"` field in [`package.json`](../../../package.json), and no `src/cli/`. G2.4 builds the `cove` binary (`dev`/`build`/`deploy`), the codegen that **installs both orphaned seams** + binds `POST /workflows/:name` to its declared handler as a distinct `kind:'workflow'` run, the load-bearing build-time validator, and the opt-in `m3` skill-packaging seam — all fail-closed before any `convex deploy`. After this phase a declared `defineAgentRegistry` is addressable by name, a `defineWorkflow` runs over HTTP, and an invalid registry is a single `[cove]` line that never reaches Convex.*

## Goal & scope

Replace flue's filesystem-module addressing + Vite/Hono build pipeline ([`@flue/cli`](../../../../flue/packages/cli/)) with a **Convex-native** authoring + wiring pipeline, and *consume* the registry seams group-1 left dormant:

1. **`cove` binary** — `cove dev` / `cove build` / `cove deploy` over the Convex toolchain (wrapping `convex dev` / `convex deploy`), with `parseArgs`/`printUsage`/dispatch, `.env` loading, and `[cove]`-prefixed diagnostics. `tsx 4.21` ([`package.json:34`](../../../package.json)) executes the TS entry.
2. **Registry loading** — a `tsx` child-import of the app-bound `convex/agentRegistry.ts` / `convex/workflowRegistry.ts` so the declared maps can be read for validation + emission **without** dragging Convex globals into the CLI process.
3. **Build-time validation** — name regex, `__coveCreatedAgent` brand, **declared-subagent-exists**, provider-resolvable, `Durability`/`Compaction` shape, workflow name/uniqueness/handler — every failure a single `[cove]` line, **fail-closed before any deploy**.
4. **Codegen that installs the orphaned seams** — a sidecar resolver module that calls `registerAgentRegistry` + `registerWorkflowRegistry`, an emitted/patched `convex/http.ts` that mounts the agent + run routes and replaces the workflow-404 stub with a `getRegisteredWorkflow` lookup + a `kind:'workflow'` run, byte-stable + content-compared so a no-op rebuild rewrites zero files.
5. **Engine wiring (the consumption half)** — point `setup.ts` at `getRegisteredAgent(name)` and add the `"workflow"` literal to `agentRequests.kind` + a `submitWorkflow` admission path so a workflow invoke is observable as a distinct run kind ([D18](../../design/07-risks-and-decisions.md)).
6. **`m3` skill packaging (opt-in)** — walk declared skill sources, `parseSkillMarkdown` each `SKILL.md`, emit a `cove.skills.json` catalog-import payload; carry the sensitive-file guards **verbatim**.

**Explicit non-goals (deferred; mirror [06 P8.5](../../design/06-phase-roadmap.md#-phase-85--cli--codegen)):**
- **`cove add` / blueprint scaffolding (m7)** — `dev`/`build`/`deploy` only; `add` is a stub printing a `[cove] … deferred (m7)` notice. flue's [`blueprint-index.ts`](../../../../flue/packages/cli/src/lib/blueprint-index.ts) (`BLUEPRINTS`/`MiniSearch`) is **not** ported.
- **Cloudflare target** — dropped ([D12](../../design/07-risks-and-decisions.md)); `--target` does not exist. Convex is the only backend.
- **Vite/Hono bundling, `flue run`/`connect`/`logs`/`docs`/`init`, the IPC harness, the event renderer** — Convex's own toolchain owns dev/deploy; the `import { type 'skill' }` attribute machinery dies with it ([08 §5](../../design/08-conventions-and-execution-boundary.md#5-dropped--obsolete-explicit)).
- **Full workflow ctx execution semantics** — G2.4 wires the route → handler → a `kind:'workflow'` run and proves the handler is *reached*; the rich `CoveContext.init()`-driven workflow body (multi-agent orchestration) is exercised by the sample agent in **[G2.5](phase-g2.5-glue-sample-otel.md)**. G2.4 builds the minimal distinct-run-kind path, not the full orchestration surface.
- **Live SDK `client.workflows.invoke`** — the native SDK workflow method is G2.1/G2.5 surface; G2.4's acceptance uses `curl POST /workflows/<name>`.

## Hardened-contract obligations

This is a **build-time/wiring** phase — it touches none of the durable-loop runtime contracts ([08 §4.1–§4.11](../../design/08-conventions-and-execution-boundary.md#4-hardened-engine-contracts)), but it must **preserve the boundaries those contracts assume** so generated wiring cannot weaken them:

- **[08 §1 Naming & namespace](../../design/08-conventions-and-execution-boundary.md#1-naming--namespace)** — every emitted + authored file uses `Cove*` brand types, `[cove]` error prefix, `cove_error`/`workflow_not_found` codes, `__coveCreatedAgent`/`__coveSkillReference` brands. No `Flue`/`flue`/`pi`-as-identifier survives in generated output. `defineAgentRegistry`/`defineWorkflow` are kept *generic verb names* (not brand-renamed) per 05, but they remain **app-bound** (next bullet).
- **[08 §1 app-bound surface](../../design/08-conventions-and-execution-boundary.md#1-naming--namespace)** / **[05 agent registry](../../design/05-public-api-and-sdk.md#agent-registry-defineagentregistry)** — `defineAgentRegistry`/`defineWorkflow` live under `convex/` and are **never** re-exported from the `@cove/runtime` barrel ([`src/runtime/index.ts`](../../../src/runtime/index.ts)), which must stay V8-safe/Convex-free. The sidecar resolvers + `src/cli/index.ts` are the install/export surface. Codegen must NOT add a runtime-barrel export — **grep-guarded in acceptance.**
- **[08 §2 Reference-header convention](../../design/08-conventions-and-execution-boundary.md#2-reference-header-convention)** — every new source file opens with the origin header (cite the flue origin from the source map + npm package where relevant). Codegen-**emitted** files additionally carry `// Auto-generated by cove (do not edit)` (the cove analogue of flue's `// Auto-generated by flue`).
- **[08 §3 Execution boundary](../../design/08-conventions-and-execution-boundary.md#3-the-execution-boundary-core-philosophy)** — generated wiring is pure orchestration glue: it registers handlers + routes and **never** touches a box, the LLM, or sandbox FS. **No `"use node"`** belongs in the emitted resolver/http.ts files (they run as `httpRouter` actions calling mutations/queries; the box is reached only from the P2/P4 engine actions, [08 §which Convex functions may touch the sandbox](../../design/08-conventions-and-execution-boundary.md#which-convex-functions-may-touch-the-sandbox)).
- **[D13 skills resolve at the call site](../../design/08-conventions-and-execution-boundary.md#skills-resolve-at-the-call-site-not-in-the-sandbox)** — `m3` packaging reads `SKILL.md` from the **host/repo at build time** to produce catalog-import rows; it must NOT emit code reading `SKILL.md` from a sandbox FS at runtime. Packaging is the *authoring* side; runtime resolution stays catalog-table.
- **[D18 distinct run kind](../../design/07-risks-and-decisions.md)** — workflow-route codegen must surface a workflow invoke as a `kind:'workflow'` run, **distinct** from an agent run, and must NOT fold workflow names into the agent registry/resolver (that would re-introduce the agentName-rekey conflation). The actual discriminator in cove is `agentRequests.kind` ([`schema.ts:168`](../../../convex/schema.ts)) — currently `prompt|skill|task|compact`; this phase adds `workflow`.

## Risks & gotchas

- **Loading the app-bound registry can drag in Convex globals.** `convex/agentRegistry.ts` imports `CreatedAgent` types and transitively the runtime; importing it in the CLI process risks pulling Convex's module globals. **Mitigation:** isolate the import in a short-lived `tsx` child (`registry-loader.ts`), run the validation *there* where the live object exists, and report diagnostics back as text. Wrap any import failure as one `[cove] failed to load <file>: <message>` line — a user typo must be a one-line error, not a 50-line stack.
- **Re-exporting registry constructs from the runtime barrel breaks the V8-safe invariant.** The most likely accidental regression is adding `export { defineAgentRegistry } from "../../convex/agentRegistry.ts"` to [`src/runtime/index.ts`](../../../src/runtime/index.ts) "for convenience." That couples the pure barrel to `convex/`. **Mitigation:** the grep-guard acceptance bar; keep all registry surface on `src/cli/index.ts` + the sidecar resolvers.
- **Wiring the workflow route + agent-by-name touches P8/P6 and the run-kind path — scope-creep risk.** `setup.ts` is model-only today; `agentRequests.kind` lacks `"workflow"`; there is no `submitWorkflow`. **Mitigation:** keep the consumption half **minimal** (task 10): the schema literal + an `admitWorkflow` peer + a `getRegisteredAgent` lookup with a fallback. The *rich* workflow body (multi-agent `ctx.init()` orchestration) is **out of scope** — proven by the G2.5 sample. Acceptance only requires the route reaches the handler as a distinct `kind:'workflow'` run.
- **`runs.kind` (design) vs `agentRequests.kind` (code).** [D18](../../design/07-risks-and-decisions.md) / [05](../../design/05-public-api-and-sdk.md#workflows-user-authored-restored) describe a `runs` table with `kind:'agent'|'workflow'`; the shipped schema uses `agentRequests.kind: prompt|skill|task|compact` ([`schema.ts:168`](../../../convex/schema.ts)). **Mitigation:** add `workflow` to the **existing** `agentRequests.kind` union (do NOT introduce a parallel `runs` table); note the naming reconciliation in the schema comment.
- **Non-deterministic codegen churns the `convex dev` watcher.** Unsorted map iteration or unstable var indexing makes byte output flip across runs, defeating the content-compare tracker and restarting `convex dev`. **Mitigation:** iterate `registry.names` in declared order, assign var indices stably (`registry-names.ts`), and content-compare before every write ([`build.ts:215–232`](../../../../flue/packages/cli/src/lib/build.ts)).
- **`cove deploy` must fail-closed.** Validation must run **before** `convex deploy` is ever spawned — a half-deployed app with an invalid registry is the failure mode. **Mitigation:** `deploy.ts` runs load→validate→codegen→tsc and short-circuits on any failure; the negative acceptance asserts (spawn spy) that `convex deploy` is never invoked.
- **Generated `convex/http.ts`/resolvers must stay box-free.** They run as `httpRouter` actions + module-load side effects; a stray `"use node"` would conflate the orchestration layer with the box layer ([08 §3](../../design/08-conventions-and-execution-boundary.md#which-convex-functions-may-touch-the-sandbox)). **Mitigation:** the emitters never write `"use node"`; the box is reached only from P2/P4 engine actions.
- **`m3` sensitive-file exclusion is a security boundary, not a nicety.** `SENSITIVE_DIRECTORIES`/`EXCLUDED_FILES`/`SENSITIVE_FILE_PATTERNS` ([`vite-import-attribute-plugin.ts:22–24`](../../../../flue/packages/cli/src/lib/vite-import-attribute-plugin.ts)) keep `.ssh`/`.aws`/`*.pem`/`credentials.json` out of a catalog payload. **Mitigation:** carry the constants + `isSensitiveFile`/`collectFiles` **verbatim**; the acceptance includes a sensitive-file-throws case.
- **`assertAgentName`/`assertUniqueNames` are file-private in cove.** Unlike the group-1 plan's assumption, [`agent-definition.ts`](../../../src/runtime/agent-definition.ts) does **not** export them (lines 285–317). **Mitigation:** route validation through `defineAgentRegistry()`/`defineWorkflowRegistry()` (which already run the regex + brand + uniqueness, [`agentRegistry.ts:25–48`](../../../convex/agentRegistry.ts)) and layer the subagent-exists + provider-resolvable + config-shape checks on top — the regex stays single-sourced, not duplicated.
- **Workflow vs agent run-kind conflation.** Emitting a workflow into the agent resolver would re-introduce the agentName-rekey conflation D18 resolved. **Mitigation:** `generate-workflow-registry.ts` writes a separate sidecar; the validator/codegen never merge the two name maps.
