# G2.3 — Channels outbound reply + ship-first adapter set
> Complete **P11**: turn the single inline `POST /channels/slack` route (inbound only, and skipping `runAuthorize`) into a shared `ChannelAdapter` pipeline — read raw bytes once → `runAuthorize` + `adapter.verify` → handshake → dedup → `mapPayload` → `submitPrompt` → persist a **reply-context** — and add the **outbound reply** that posts the terminal result back *after* the run finalizes (never inside the ~3s ack window). Port the eight ship-first channels (Slack, Discord, GitHub, Teams, Telegram, Google Chat, Linear, Notion) against that one contract with **zero engine/loop edits**. Design-of-record: [06 — Roadmap, Phase 11](../../design/06-phase-roadmap.md) + [05 — Public API & SDK](../../design/05-public-api-and-sdk.md), [08 §2/§3/§5](../../design/08-conventions-and-execution-boundary.md). Decisions: [D1 (SQL adapters obsolete), D3 (pluggable authorize), D14 (reuse `agentRequests`)](../../design/07-risks-and-decisions.md). Prior plan: [group-1/phase-11](../group-1/phase-11-channels.md).

## Goal & scope

P11 shipped only its **pure inbound prerequisite** for **one** channel: [`src/runtime/channels/slack.ts`](../../../src/runtime/channels/slack.ts) (`verifySlackSignature`/`parseSlackPayload`/`slackSessionRef`/`slackDedupKey`), the shared dedup mutation [`convex/channels.ts`](../../../convex/channels.ts) `markWebhookSeen`, and an **inline** `POST /channels/slack` `httpAction` in [`convex/http.ts:144-191`](../../../convex/http.ts). Three gaps remain, all live in this phase:

1. **No outbound.** The inline route admits a run and returns `"ok"` — the result is never posted back. There is no `reply.ts`, no reply-context on the request row, and `runHandler` has no post-finalize trigger. flue posted replies from a long-lived process; cove has no equivalent yet.
2. **The Slack route skips `runAuthorize`.** Every other transport ([`convex/http.ts`](../../../convex/http.ts) `/agents/`, `/runs/`) calls `await runAuthorize(ctx, req)`; the channel route ([`convex/http.ts:147-191`](../../../convex/http.ts)) does **not** — a real gap against [D3](../../design/07-risks-and-decisions.md) / [06 P11](../../design/06-phase-roadmap.md) ("the `authorize` hook gates inbound just like any other transport").
3. **No shared contract, seven channels missing.** The Slack inbound logic is bespoke; the seven other ship-first channels (Discord, GitHub, Teams, Telegram, Google Chat, Linear, Notion) are not ported. flue's per-package logic lives in [`flue/packages/{discord,github,teams,telegram,google-chat,linear,notion}/src/`](../../../../flue/packages/).

G2.3 closes all three. Four deliverable groups, all transport-side (no LLM decision, no box — channel `httpAction`s admit + reply, they never run the loop, per [08 §3](../../design/08-conventions-and-execution-boundary.md#3-the-execution-boundary-core-philosophy)):

1. **`convex/channels/{inbound,dedup,reply}.ts`** — the shared 4-step inbound pipeline parameterized by a `ChannelAdapter`, the `(provider, eventId)` dedup helper (promoted from `markWebhookSeen`), and the outbound reply dispatch fired **after** the run terminalizes.
2. **Reply-context on `agentRequests`** — a new optional `replyContext` field (provider + addressing tuple + `response_url` + target) captured at admission so a finished run can address its reply when the webhook request object is long gone.
3. **Post-finalize reply trigger** — `runHandler` (or `finalize`) schedules `reply.dispatch` exactly once when the request reaches a terminal status, replay-idempotent (a journaled `step.run*` after the loop, **not** a second top-level hook).
4. **The eight ship-first adapters** — `convex/channels/slack/index.ts` (refactor the inline route) + `convex/channels/{discord,github,teams,telegram,google-chat,linear,notion}/index.ts`, each a `ChannelAdapter{name,verify,mapPayload,postReply}`; Teams + Google Chat add a `"use node"` JWT/JWKS `auth.ts`. `convex/channels/index.ts` carries the `channelRegistry` + deferred-channel stub notes.

**In scope:** the shared pipeline, the reply-context column + trigger, the eight adapters, the `authorize` gate on inbound, per-channel verify/dedup/map/reply tests on mocks (no live providers), and the SQL-adapter-collapse confirmation (D1). **Out of scope (deferred / non-goals):**
- **Streaming "typing…" replies** (soft-depends on G2.1's reactive `events`). This cut posts the **final result only**, read from the terminal `agentRequests` row — the minimum P11 bar. ([group-2 README open question 7](README.md#open-questions-surface-before-committing).)
- **Deferred channels** (Intercom, Resend, Salesforce Marketing Cloud, Zendesk) and **TBD** (Messenger, WhatsApp, Twilio, Shopify, Stripe) — documented as stubs in `channels/index.ts`, not built ([06 P11 scope](../../design/06-phase-roadmap.md)).
- **Any engine/loop/SDK edit.** The reply trigger is the *one* touch to `runHandler`/`finalize`, and it adds a journaled call — it changes no loop semantics. The shared-contract acceptance bar (≥2 more channels via adapter only) is proven by a **zero engine diff**.
- **SQL storage adapters** (`postgres`/`libsql`/`mysql`/`mongodb`/`redis`) — already collapsed (D1); **no adapter code exists** and none is written. This phase only *confirms* the drop is cited.
- **Discord interaction *responses*** beyond PING ack + deferred-then-followup (we ack the interaction synchronously and post the agent reply via the bot REST API / `response_url`, like every other async channel).

## Dependencies

- **P8 (HTTP + auth) — LANDED.** Channels are `httpAction`s on the P8 [`httpRouter`](../../../convex/http.ts) (`http` in `http.ts:35`); inbound calls [`api.invoke.submit.submitPrompt`](../../../convex/invoke/submit.ts) (`submit.ts:10-31`); [`runAuthorize(ctx, req)`](../../../convex/auth.ts) (`auth.ts:19-22`) gates them; [`renderHttpError`](../../../src/runtime/http.ts) (`http.ts:71`) + the `CoveHttpError` subclasses (`UnauthorizedError` at `http.ts:38`, `InvalidJsonError` at `http.ts:28`) render failures onto the `CoveApiError` envelope. The webhook ack (200/202) uses the same helpers.

The reply path reads the terminal `agentRequests` row this admission produced.
- **P4 (Durable engine) — LANDED.** [`engine/runHandler.ts`](../../../convex/engine/runHandler.ts) `agentRun` ends every run with `step.runMutation(internal.engine.finalize.run, …)` (`runHandler.ts:40-49`); [`engine/finalize.ts`](../../../convex/engine/finalize.ts) `run` terminalizes the request (`finalize.ts:34-45`). The post-finalize reply trigger extends exactly this seam — a journaled `step.run*` after `runAgentLoop` returns. **The `task()` delegation pattern is the precedent:** [`engine/task.ts`](../../../convex/engine/task.ts) already starts a child workflow + polls a terminal snapshot (`createChildRequest`/`getChildResult`, `task.ts:36-118`) from within the engine — the reply trigger is a strictly simpler journaled post-finalize call.
- **P5 (SOR / dedup) — LANDED.** [`schema.ts`](../../../convex/schema.ts) `meta` table + `by_key` index (`schema.ts:396-399`); `markWebhookSeen` (`channels.ts:10-21`) already inserts a `meta` dedup row idempotently. `agentRequests` (`schema.ts:163-239`) is the reuse target — **no new inbound table** (D14).
- **G2.1 (Reactive events) — soft, deferred-use only.** The streaming-reply option reads G2.1's `events` stream; **this cut does not** — it polls/reads the terminal `agentRequests` row. G2.3 ships without G2.1.

No hard upstream group-2 dep. G2.3 is parallelizable with G2.2 (MCP); both soft-depend on G2.1.

**External installs:** none hard for offline CI — Web Crypto (`crypto.subtle`) covers Slack/GitHub/Notion/Linear HMAC, Discord Ed25519, and Telegram secret-token digest, all already used by the ported flue routes. **Teams + Google Chat JWT verification needs `jose`** (flue's `teams`/`google-chat` use `jose@6.2.3`); add it as a dependency, imported **only** under `convex/channels/{teams,google-chat}/auth.ts` (`"use node"`). Live e2e (not offline CI) needs: Slack bot token / a captured `response_url`, `GITHUB_WEBHOOK_SECRET`, `DISCORD_PUBLIC_KEY`, `TELEGRAM_SECRET_TOKEN`, Teams `appId` + Google Chat audience/project-number, and the provider REST tokens for `postReply`.

## Hardened-contract obligations

- **[D14 — reuse `agentRequests` + HTTP submit](../../design/07-risks-and-decisions.md).** Channels land through the **existing** `submitPrompt` admission into `agentRequests`; **no separate inbound table**. The four steps are fixed (verify → dedup → `submitPrompt` → reply), and the reply-address state rides on the request row (`replyContext`), not a bespoke store. The inbound `meta` dedup row is the *only* added row, and it already exists.
- **[D3 / 08 §3 — `authorize` gates inbound](../../design/07-risks-and-decisions.md).** `verifyThenAdmit` calls `runAuthorize(ctx, req)` **before** `adapter.verify` and before any admission — closing the current gap where `/channels/slack` ([`http.ts:147-191`](../../../convex/http.ts)) skips it. Channel-secret verification is the *channel's* step; `runAuthorize` is the *framework* gate, and both must pass. A rejected authorize never admits.
- **Per-channel signature verification (security), [08 §3 boundary](../../design/08-conventions-and-execution-boundary.md#3-the-execution-boundary-core-philosophy).** Every inbound route verifies the provider signature over the **raw bytes** before admission — Slack HMAC + window, GitHub `X-Hub-Signature-256`, Discord Ed25519, Telegram secret-token digest, Linear HMAC + window, Notion HMAC, Teams/Google-Chat JWT. A failed verify → 401 via `renderHttpError`, **no** admission. The `httpAction` admits + (later) replies; it never runs the loop or touches the box.
- **Idempotent dedup, dedup precedes admission.** A replayed webhook (same `(provider, eventId)`) is a **no-op 200** via `markWebhookSeen` — and the dedup insert runs **before** `submitPrompt`, so a replay never spawns a second run between admission and the dedup check.
- **Reply is async, after-terminal, exactly-once.** The provider ack window (Slack ≤3s, etc.) forbids `?wait=result` inside the handler. The handler acks immediately; `reply.dispatch` posts **only** after `finalize` terminalizes, fired from a journaled post-`runAgentLoop` `step.run*` (replay-idempotent via the `repliedAt` stamp). **Zero** posts before terminal, **exactly one** after.
- **JWT/JWKS is `"use node"` + cached + claim-checked (security-sensitive).** Teams + Google Chat `auth.ts` run under `"use node"` (jose + network discovery), cache keys per-process (re-discovered on cold actions — **not** assumed durable), and enforce issuer/audience/identity + RS256 + `kid` exactly as flue does. The barrel/V8 isolate stays clean (jose imported only there).
- **[08 §2 — reference-header convention](../../design/08-conventions-and-execution-boundary.md#2-reference-header-convention).** Every new file opens with a header citing its flue origin (`@flue/<channel>` + `packages/<channel>/src/<file>.ts`) and the cove package it exports as, plus the key transform (Hono dropped, raw-bytes-once, per-process cache caveat).
- **[D1 — SQL adapters obsolete](../../design/07-risks-and-decisions.md).** One Convex adapter; the SQL backends are dropped, not ported, and the drop is cited ([08 §5](../../design/08-conventions-and-execution-boundary.md#5-dropped--obsolete-explicit)). No code.

## Implementation tasks

1. **Define the contract** — `convex/channels/types.ts`: `ChannelAdapter`, `ReplyContext`, `SubmitSpec` (`{ message, instanceId, sessionName, eventId, replyContext, model? }`), `VerifyResult` (`{ ok: true } | { ok: false; status: number }`), and the `mapPayload` return union (`{ kind:'submit'; spec } | { kind:'handshake'; body } | { kind:'ignore' }`). Pure types only (no Convex import) — V8-safe, importable by both the route and the tests.
2. **Promote dedup** — `convex/channels/dedup.ts`: `dedupKey(provider, eventId)` (generalize `slackDedupKey`, `slack.ts:90-92`) + re-export `markWebhookSeen` (`channels.ts:10-21`) unchanged. Keep the `meta`/`by_key` ledger (no new table). Decide retention: rely on idempotent insert; note a TTL-cleanup follow-up if `meta` growth matters.
3. **Build the shared pipeline** — `convex/channels/inbound.ts` `verifyThenAdmit(ctx, adapter, req)`:
   1. `const rawBody = await req.text()` **once** (signature needs raw bytes — re-reading/re-serializing breaks HMAC/Ed25519).
   2. `await runAuthorize(ctx, req)` (framework gate; **the closed gap**).
   3. `const v = await adapter.verify(ctx, req, rawBody)`; on `!v.ok` → `renderHttpError(new UnauthorizedError(...))` (or the verify's status), **no admission**.
   4. `const m = adapter.mapPayload(JSON.parse(rawBody))`; `handshake` → return its body; `ignore` → `new Response("ok")`, **no admission**.
   5. `const { isNew } = markWebhookSeen({ key: dedupKey(adapter.name, m.spec.eventId) })`; `!isNew` → `new Response("ok (duplicate)")`.
   6. `await submitPrompt({ ...m.spec, replyContext: m.spec.replyContext })`; ack `new Response("ok")`.
   Wrap in `try/catch → renderHttpError` exactly like the existing routes.
4. **Schema + admission threading** — `convex/schema.ts`: add `replyContext`/`repliedAt` to `agentRequests`. `convex/invoke/submit.ts`: add `replyContext` arg; `convex/invoke/admit.ts`: patch it onto the inserted row (`admit.ts:108-121`). Native/HTTP runs pass no `replyContext` (reply is a channel-only concern).

The `repliedAt` guard is the replay double-post guard.
6. **Post-finalize trigger** — `convex/engine/runHandler.ts`: after `runAgentLoop` returns (and finalize has run), add `await step.runAction(internal.channels.reply.dispatch, { requestId })` as a **journaled** step (idempotent: a replay re-runs it, the `repliedAt` stamp makes it a no-op). This is the **only** engine touch and adds no loop semantics. (Decision vs `scheduler.runAfter`: the journaled in-workflow call is preferred — `runAfter` is fire-and-forget and not journaled, risking a lost or doubled post on replay; see Risks.)

## Acceptance

Start from [06 P11's bar](../../design/06-phase-roadmap.md) — *at least one channel (Slack) verifies an inbound event, dedups a replayed webhook, drives a run via `submitPrompt`, and posts the reply back* — sharpened:

1. **Slack end-to-end on mocks.** A valid signed `event_callback` → `submitPrompt` admits → the run terminalizes → `reply.dispatch` calls the **mocked** `postReply` to the **captured `replyContext`** (channel + `thread_ts`) **exactly once**. A **forged** signature → 401, and a grep/assert shows **no** `submitPrompt` call. A **replayed** delivery (same `event_id`) → exactly **one** run (the second is a dedup no-op 200). `url_verification` → challenge echo with **no** admission. *(Test: `convex/channels/__tests__/slack.test.ts`.)*
2. **Reply is async + exactly-once.** Asserted: **zero** `postReply` invocations before the request reaches a terminal status, **exactly one** after; a forced re-run of `reply.dispatch` (simulating workflow replay) with `repliedAt` set is a **no-op** (no second post). The handler returns its ack without awaiting the run. *(Test: `slack.test.ts` + `inbound.test.ts`.)*
3. **Shared contract holds for ≥2 more channels via adapter only.** GitHub (`X-Hub-Signature-256` HMAC) **and** one JWT channel (Teams or Google Chat, with an injected key stub) pass the same verify→dedup→submit→reply path using only their adapter. **The diff touches no engine/loop file** beyond the single `runHandler` reply-trigger line shared by all channels — `git diff --stat` over `convex/engine/` shows only `runHandler.ts` (one journaled call), proving no per-channel engine work. *(Test: `github.test.ts` + `teams.test.ts`.)*
4. **`authorize` gates inbound.** An inbound call (Slack + ≥1 other) whose installed `authorize` hook throws **never admits** — both the channel-secret verify and the framework gate are enforced, and the gate runs *before* admission. A grep confirms `verifyThenAdmit` calls `runAuthorize`, closing the `http.ts:147` gap. *(Test: `inbound.test.ts`.)*
5. **No new inbound table (D14).** Runs land in `agentRequests`; the reply address is `agentRequests.replyContext`; dedup is the `meta` `by_key` row. `git diff convex/schema.ts` adds **only** the `replyContext`/`repliedAt` columns — **no `defineTable`** for inbound.

## Risks & gotchas

- **The ~3s ack window vs run latency — reply MUST be async.** A full agent run far exceeds Slack's ≤3s ack. **Do not** `?wait=result` inside the webhook handler. The handler acks immediately (`"ok"`); `reply.dispatch` posts when the run terminalizes. *Mitigation:* acceptance bar 2 asserts zero pre-terminal posts and the handler never awaits the run.
- **Reply-trigger mechanism + replay-idempotency (cove has no top-level onComplete hook).** A workflow **replay** must not double-post. *Mitigation:* fire `reply.dispatch` as a **journaled** `step.runAction` after `runAgentLoop` (replayed deterministically) **and** gate it with the `repliedAt` stamp so even a re-dispatch is a no-op. The `task()` precedent ([`task.ts:99-104`](../../../convex/engine/task.ts)) shows journaled post-admission engine calls are the sanctioned pattern. *Avoid* `scheduler.runAfter` as the primary trigger — it is fire-and-forget, **not** journaled, and can be lost or doubled across replay.
- **Signature verify needs RAW bytes — read body once.** HMAC/Ed25519 run over the exact request bytes, not re-serialized JSON. *Mitigation:* `verifyThenAdmit` does `await req.text()` **once** and hands the same `rawBody` string to `adapter.verify` *before* `JSON.parse`. (The inline route already does this, `http.ts:149`; the shared pipeline must preserve it for every channel.)
- **JWT/JWKS for Teams/Google-Chat is security-sensitive `"use node"` work.** Skipping issuer/audience/identity checks, or trusting an unverified `kid`, is a forgery hole. *Mitigation:* port flue's verifiers **near-verbatim** (issuer + audience + `RS256` + `kid` + `serviceurl`/identity claims, `teams/auth.ts:79-95`, `google-chat/auth.ts:110-156`); cache keys **per-process only** (re-discover on cold actions — not durable); inject `fetch` so tests run offline with a stub JWKS. jose imported **only** there.
- **Per-process cache is not durable across actions.** Teams/Google-Chat key caches live in module scope of a `"use node"` action and die with the isolate. *Mitigation:* treat re-discovery on a cold action as normal (the cache is a hot-path optimization, not a correctness dependency); the unknown-kid/key refresh cooldown (`teams/auth.ts:9,118`, `google-chat/auth.ts:16,68`) bounds discovery storms.
- **Payload heterogeneity → loops if mapped greedily.** Discord interactions, GitHub event types, Telegram updates, and Notion events carry many non-message variants. *Mitigation:* every `mapPayload` has an explicit **ignore branch** (return `{kind:'ignore'}`) for handshakes, non-text, non-comment, and bot/self events — never admit an event the bot itself produced.
- **Bot-echo self-reply loops.** Posting a reply can re-trigger the same channel's inbound (the bot's own message arrives as an event). *Mitigation:* per-channel self-message guard in `mapPayload` (Slack's `!e.bot_id`/`subtype === undefined`, `slack.ts:66-71`, is the template); GitHub/Linear filter on the actor/bot id; Discord/Telegram ignore the bot's own author id.
- **Don't port the Hono/DS plumbing.** flue's routes assume the Hono `flue()` app + `Context`/`Handler`/`bodyLimit` streaming reader. *Mitigation:* port only the **verify + map + reply** logic into plain `httpAction`s; `req.text()` replaces the manual `readBody` byte-limit reader (Convex bounds the body).
- **The reply trigger is the one engine touch — keep it semantics-free.** A careless edit to `runHandler` could change loop behavior and break the "no engine work" bar. *Mitigation:* the trigger is a single appended `step.runAction` *after* the loop+finalize; it reads nothing the loop depends on and changes no control flow. Acceptance bar 3's `git diff --stat` over `convex/engine/` proves it.
- **Unknown / deferred channel path.** A `POST /channels/intercom` (deferred) must not 500. *Mitigation:* the `/channels/` route returns a clean 404 (`renderHttpError`) for any name absent from `channelRegistry`; the deferred set is documented as stub notes.
