# Phase 7 — HITL approval gate

> Park an approval-gated tool call on a durable workflow event, resume on an idempotent fail-loud `submitApproval`, re-validate approver-edited args, and terminalize a long-parked run as `cancelled`. Design-of-record: [06 — Roadmap](../design/06-phase-roadmap.md) + [04 — Durable Engine](../design/04-durable-engine.md#hitl--the-approval-state-machine) + [08 §4.4](../design/08-conventions-and-execution-boundary.md#44-hitl-state-machine) + [03 — Data Model](../design/03-data-model-sor.md#approvals--hitl-new). Decisions: [D1–D19](../design/07-risks-and-decisions.md) (esp. [D4](../design/07-risks-and-decisions.md)).

## Goal & scope

Human-in-the-loop is a **new additive capability** ([D4](../design/07-risks-and-decisions.md)) — flue has no equivalent; it is natural on a durable workflow. This phase adds the **approval gate** between the model's decision and tool execution:

- A tool flagged `isHitl` (or a session/skill policy that requires approval) **parks** the run on a durable workflow event instead of executing immediately. While parked **no compute burns** — the workflow is suspended in the journal.
- A host calls **`submitApproval`** (approve / approve-with-edits / reject); the mutation flips the `approvals` row and **emits the workflow event**, waking the run.
- The state machine in [08 §4.4](../design/08-conventions-and-execution-boundary.md#44-hitl-state-machine) is honored exactly: globally-unique event key, idempotent fail-loud submit, durable/queued event (no lost wakeups), re-validated approver-edited args (a `ToolInputValidationError` → error tool-result, not a crash), and timeout-while-parked → `cancelled`.

**In scope:** the `approvals` mutations (`submitApproval`, plus internal `parkApproval`/`applyApproval`/queries), the durable `awaitEvent` gate wired into `runHandler.agentRun`, the `isHitl` tool flag + a minimal approval **policy** resolver at `setup`, the per-approval timeout, and the caller-facing `submitApproval` surface on the SDK/harness facade plus an `awaitingApproval` reactive read.

**Out of scope (deferred, recorded so callers are not surprised):** general in-turn **steering / follow-up queues** and `beforeToolCall`/`afterToolCall` hooks ([D17](../design/07-risks-and-decisions.md), [08 §5](../design/08-conventions-and-execution-boundary.md#5-dropped--obsolete-explicit)) — HITL covers only the approval use case. Rich approval-card UI components (a P9 `@cove/react` concern). HTTP `submitApproval` route (the route wiring lands with the HTTP surface in P8; this phase ships the mutation + native/SDK entry, and the route is a thin P8 add-on over the same mutation).

## Hardened-contract obligations

This phase exists to honor **[08 §4.4](../design/08-conventions-and-execution-boundary.md#44-hitl-state-machine)** in full, and must not break two adjacent contracts:

1. **§4.4 — Globally-unique event key.** The event name is exactly `` `approval:${requestId}:${toolCallId}` `` — unique across requests *and* tool calls. (`requestId` is the `agentRequests` `_id`; `toolCallId` is the LLM-issued string.) The `approvals` row's `by_toolCall` index plus `(requestId, status)` enforce that no two pending approvals collide.
2. **§4.4 — Idempotent, fail-loud submit.** `submitApproval` **rejects** (throws `ApprovalNotPendingError`) if `approvals.status !== "pending"`. A double-submit cannot flip an already-resolved approval; the second caller gets a typed error, never a silent no-op and never a second `sendEvent`.
3. **§4.4 — Durable / queued event (no lost wakeups).** The gate awaits **by name** (`ctx.awaitEvent({ name, validator })`), which the component implements by creating a queued `"awaited"` event when none exists. Therefore a `submitApproval` that fires its `sendEvent` *before* the workflow reaches the await is **delivered when the await runs** — the wakeup is never dropped. (Do **not** use `createEvent`-then-await-by-id in a way that requires the event to pre-exist before submit; by-name await is the lost-wakeup-proof form.)
4. **§4.4 — Approver-edited args re-validated.** Edited args run through `normalizeToolDefinition`'s `v.safeParse` in `dispatchTools`; a failure surfaces as a `ToolInputValidationError` **error tool-result** (the model self-corrects), never a step crash. ([08 §4.5](../design/08-conventions-and-execution-boundary.md#45-tool-rebuild-from-frozen-descriptors) "a buildTools/validation failure becomes an error tool-result, never a step crash" applies here verbatim.)
5. **§4.4 — Timeout while parked → cancelled.** A request parked beyond `durability.timeoutMs` terminalizes `cancelled` (`cancelReason: "timeout"`), not failed and not hung forever.
6. **§4.1 — Replay determinism (must not break).** The park/await beat is a journaled `step` checkpoint; on replay the workflow reads the recorded `awaitEvent` outcome and re-takes the same branch — it must **not** re-park or re-emit. `applyApproval` and the rejection/edit tool-result must be idempotent (keyed on `toolCallId`, replace-in-place) exactly like `appendToolResult`, so a replayed dispatch never double-acts.
7. **§4.3 — Cancel short-circuit interplay.** If the run is cancelled (host abort) while parked, the awaiting workflow is torn down by `workflow.cancel`; the pending `approvals` row is marked terminal (`rejected`/`cancelled`-decision) so a late `submitApproval` finds a non-`pending` status and fail-louds. The abort sequence stays atomic ([08 §4.3](../design/08-conventions-and-execution-boundary.md#43-cancel-short-circuit)).

## Acceptance

Start from [06 P7](../design/06-phase-roadmap.md#-phase-7--hitl)'s bar; each invariant gets a concrete test, not prose.

- [ ] **Parks without burning compute.** A run whose model calls an `isHitl`-flagged tool writes an `approvals` row `status: pending`, patches `agentRequestSteps.toolCalls[].isHitl = true`, and **suspends** at `awaitEvent` — no `dispatchTools` execute fires, the request stays `running`-parked, and the workflow journal shows the run blocked on the event (no busy loop, no repeated `llmStep`).
- [ ] **Approve resumes correctly.** `submitApproval({ approvalId, decision: { kind: "approved" } })` flips the row to `approved`, emits the event, the workflow wakes, `dispatchTools` executes the tool, and the run reaches a terminal `completed`.
- [ ] **Reject resumes with a rejection tool-result.** `submitApproval({ … decision: { kind: "rejected", reason } })` resumes the run with an **error tool-result** carrying the reason (tool **not** executed); the model can self-correct and the loop continues.
- [ ] **Idempotent, fail-loud double-submit.** A second `submitApproval` on the same approval throws `ApprovalNotPendingError` and emits **no** second event; the run is unaffected (the wakeup already happened exactly once).
- [ ] **No lost wakeup (race).** A `submitApproval` that fires **before** the workflow reaches `awaitEvent` is **not lost** — the by-name queued event is matched when the await parks, and the run resumes. (Test by sending the event immediately after admission, before the gate is reached.)
- [ ] **Edited args re-validated.** `submitApproval({ … decision: { kind: "approved", editedArgs } })` with schema-conformant edits executes with the edited args; with **schema-violating** edits, the run gets a `ToolInputValidationError` **error tool-result** (not a crash, not a silent execute).
- [ ] **Survives redeploy + timeout → cancelled.** A parked run survives a simulated redeploy (the durable wait persists in the journal); a run parked beyond `durability.timeoutMs` terminalizes `cancelled` with `cancelReason: "timeout"` (the `approvals` row marked terminal), and a subsequent `submitApproval` fail-louds.
- [ ] **Replay determinism.** A crash-after-`applyApproval` replay re-takes the recorded branch — it does **not** re-park, re-emit, or re-execute the tool (idempotent `appendToolResult` keyed on `toolCallId`); honoring [08 §4.1](../design/08-conventions-and-execution-boundary.md#41-llmstep-replay-determinism-critical).
- [ ] **Green build.** `tsc --noEmit` exits 0; the suite passes; all new files carry the reference header.

## Risks & gotchas

- **`waitForEvent` is a doc placeholder — the real API is `ctx.awaitEvent` / `workflow.sendEvent`.** The design pseudo-code ([04](../design/04-durable-engine.md#the-loop)) writes `step.waitForEvent(...)`; the `@convex-dev/workflow@0.3.x` `WorkflowCtx` actually exposes `awaitEvent({ name, id?, validator? })` and the manager exposes `sendEvent(ctx, { workflowId, name|id, validator, value })` + `defineEvent`/`createEvent`. Build against the real surface (verified in `node_modules/@convex-dev/workflow/dist/client/`), not the placeholder name.
- **By-name await is the lost-wakeup-proof form.** Awaiting by **name** auto-creates a queued `"awaited"` event if none exists, so a pre-await `sendEvent` is matched — this is the §4.4 "durable/queued" guarantee. Awaiting by a pre-created **id** requires the event to exist first and risks a race; prefer the by-name form. The **validator must match exactly** on `awaitEvent` and `sendEvent` or the component rejects the send.
- **`sendEvent` runs from a mutation, not an action.** `submitApproval` is a Convex **mutation** (it touches no box, only tables + the workflow component) — correct per the boundary ([08 §3](../design/08-conventions-and-execution-boundary.md#which-convex-functions-may-touch-the-sandbox): `invoke/*` are box-free mutations). Do **not** make it an action; `workflow.sendEvent` needs a `RunMutationCtx`.
- **Timeout race via journaled `sleep`.** Racing `awaitEvent` against `ctx.sleep(timeoutMs)` is the durable timeout pattern, but **both are journaled `step` awaits** — `Promise.race` records whichever resolves first into the journal, so replay re-takes the same winner. Do not implement the timeout with a wall-clock `setTimeout` in an action (it would not survive a redeploy and would be non-deterministic on replay).
- **The event value crosses the journal — keep it small & serializable.** `ApprovalDecision` (including `editedArgs`) is the `sendEvent` `value` and is journaled. It must be plain JSON (no closures, no `Date` objects — use epoch ms). Edited args are re-validated downstream, so keep them as raw `Record<string, unknown>`.
- **Self-cancel from inside the parked workflow.** The timeout path calls `workflow.cancel(self)` from within the running workflow's own helper — make sure the gate helper signals the `runHandler` to stop (return a terminal sentinel / throw a caught terminal error) rather than falling through to `dispatchTools` after cancelling itself.
- **Idempotency on replay.** `parkApproval` must no-op if the row exists for `(requestId, toolCallId)`; `applyApproval` and the rejection/edited tool-result must replace-in-place by `toolCallId` (mirror `appendToolResult`). A replayed gate beat must not insert a duplicate `approvals` row, re-emit, or double-execute.
- **`isHitl` rides the frozen descriptor, not the model schema.** The flag is on the `ToolDefinition` and travels onto `plan.tools` (frozen at setup) — it must **not** leak into the JSON-Schema the model sees (the model has no notion of approval). `agentRequestSteps.toolCalls[].isHitl` is set by the *engine* (from the policy), not by the model.
- **Parked is a sub-state of `running`.** Per [`schema.ts`](../../convex/schema.ts) the `agentRequests.status` enum has no separate `parked`/`awaiting` value — a parked run stays `running`.
